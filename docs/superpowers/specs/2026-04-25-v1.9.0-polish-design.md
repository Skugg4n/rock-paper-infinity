# Design — v1.9.0 Polish Release

**Date:** 2026-04-25
**Target version:** v1.9.0
**Status:** First pass. The user has a separate change document to integrate later.
**Vision reference:** [vision.md](../../../vision.md)

This release converts Rock, Paper, Infinity from a working v1.8.0 prototype into something stable and polished enough to share with test users. It introduces the chapter-card transition system that makes the game's structure visible, reworks Phase 2 progression so unlocks always have a tease, hardens save/load against corruption, and ships the result on GitHub Pages.

The release does **not** add new content (no Code Processor, no Phase 3 work, no accessibility pass). Those are deferred. It also does not add onboarding, tutorials, or hints — those are explicit non-goals per [vision.md](../../../vision.md).

## Goals

1. Make the game's chapter structure visible through the bombastic chapter-card transition (vision §"Chapter transitions").
2. Eliminate "unlock kaos" in Phase 2 — every unlock has a preceding tease, no two unlocks fire on the same threshold.
3. Bring text discipline to the level the vision asks for (icons over text), applied consistently rather than minimally in some places.
4. Make endgame animations reliable and pleasant — Phase 2 buildings specifically.
5. Make save/load survive corruption silently.
6. Ship to a stable hosting target so test users can play.
7. Update all the docs/metadata that have drifted from the actual state of the code.

## Non-goals

- Phase 3 (WAR) implementation. The chapter card plays a `III · WAR` placeholder and the screen ends on the *to come* wall.
- Phase 4 (ESCAPE) anything.
- Code Processor upgrade (deferred from PROJECTPLAN).
- Accessibility (keyboard nav, ARIA) — deferred.
- Tutorials, hints, first-run popups — explicit vision non-goal.
- Audio / music / SFX — out of scope.
- Build system / Vite migration — out of scope.

## Part 1 — Chapter card system

A reusable overlay component that plays the bombastic transition described in vision §"Chapter transitions". Used at three points in v1.9.0: game start (`0 → I`), Bank commit (`I → II`), and 50k pop (`II → III placeholder`).

### Module: `src/chapterCard.js`

Single file. Exports:

```js
playChapterCard({ roman, title, mode, onMidpoint }) → Promise<void>
```

- `roman` — string, e.g. `"I"`, `"II"`, `"III"`.
- `title` — string, e.g. `"TRIVIAL"`, `"WAR"`. Always rendered uppercase regardless of input.
- `mode` — `"normal"` (default, fade out white to next scene) or `"to-come"` (fade to black, show *to come* in lighter weight, hold indefinitely).
- `onMidpoint` — optional callback fired at the start of the hold phase (~600 ms in). This is where Phase 1 teardown and Phase 2 init run on the I→II trigger.

Returns a Promise that resolves when the animation has fully played out. For `mode: "to-come"`, the promise never resolves (the game stops there).

### DOM

A single fixed-position overlay appended to `<body>`:

```html
<div id="chapter-card" class="chapter-card" aria-hidden="true">
  <div class="chapter-card__veil"></div>
  <div class="chapter-card__content">
    <div class="chapter-card__roman">I</div>
    <div class="chapter-card__title">TRIVIAL</div>
    <div class="chapter-card__suffix" hidden>to come</div>
  </div>
</div>
```

The overlay covers the viewport. While active it blocks all input (`pointer-events: auto` on the veil). When idle it sits at `pointer-events: none; opacity: 0`.

### Animation timing (matches vision)

| Phase | Duration | What happens |
|-------|----------|--------------|
| 1 | 300 ms | Veil opacity 0 → 1 (white). Content opacity stays 0. |
| 2 | 300 ms | Roman + title opacity 0 → 1. |
| 3 | 1000 ms | Hold. `onMidpoint` fires at the start of this phase. |
| 4 | 300 ms | Roman + title opacity 1 → 0. |
| 5 | 300 ms | Veil opacity 1 → 0. |

Total: 2.2 s.

For `mode: "to-come"`:
- Phase 4 plays as normal (title fades out).
- Phase 5 is replaced: veil transitions from white to black (300 ms), then *to come* fades in below where the title was (300 ms, lighter weight, smaller).
- The promise never resolves. Pointer-events stay blocked indefinitely.

### Typography

`@import url('https://fonts.googleapis.com/css2?family=Bebas+Neue&display=swap');` added to `index.html` `<head>`. Bebas Neue chosen because it's free, hosted on Google Fonts, and matches the vision spec (bold condensed block sans-serif).

The chapter card is the **only** place this font is used. Everywhere else stays on the system stack.

### Triggers

**Game start (`0 → I`).** `main.js` checks `localStorage` on load: if no `rpi-save` and no `rpi-stage2` exist (i.e. truly fresh player), it plays `playChapterCard({ roman: "I", title: "TRIVIAL" })` before calling `setPhase()`. Returning players skip the card entirely.

**Bank commit (`I → II`).** Replace the current direct call to `setPhase(phases.CITY)` in the Bank handler with:

```js
playChapterCard({
  roman: "II",
  title: "CAPITAL",
  onMidpoint: () => setPhase(phases.CITY),
});
```

The hold phase covers the teardown/init flicker.

**50k pop (`II → III placeholder`).** Phase 2 already has logic that fades in the competitor island at 50k. Move that fade-in down to **40k pop** (see Part 2). At 50k pop, fire:

```js
playChapterCard({
  roman: "III",
  title: "WAR",
  mode: "to-come",
  onMidpoint: () => disableSaving(),  // game is paused here
});
```

The promise never resolves. The game stops on the black wall.

**Past-threshold load.** If a Phase 2 save is loaded with population already ≥ 50k (e.g. the player closed the tab on the WAR wall and came back), Phase 2 `init()` triggers the same card immediately with `onMidpoint: () => disableSaving()`. The player lands directly on the wall, as if the chapter card had been frozen mid-air waiting for their return. No delayed gameplay, no "where's the wall?" moment.

### Race / safety

- A module-level boolean `_cardActive` prevents the card from being triggered twice. Subsequent calls during animation are ignored.
- The `onMidpoint` callback is wrapped in a try/catch — a thrown error logs but does not break the animation.
- `requestAnimationFrame` drives the timing, not `setTimeout`. The card cannot be skipped by `setInterval` drift or background-tab throttling.

### Tests (`__tests__/chapterCard.test.js`)

Pure JS tests that don't render. Use jsdom (already configured by Jest):

- `playChapterCard` resolves after ~2200 ms in normal mode.
- `playChapterCard` with `mode: "to-come"` never resolves (use `Promise.race` with a timeout to assert).
- `onMidpoint` fires roughly at hold start.
- A second call while the first is active is a no-op.

## Part 3 — Text discipline

The vision says icons over text. The codebase has drifted in two directions:

1. **Phase 1 has tooltip infrastructure but `upgradeDescriptions = {}` is empty.** Tooltips render only the cost, no description. So tooltips are effectively partial.
2. **Phase 2 has tooltip strings with `Unlock` prefixes and parenthetical numbers** (`(industry: X.X)`).

### Decision

**Drop tooltip-text descriptions everywhere. Keep only icon + cost (Roman ≥10).** This is consistent across Phase 1 and Phase 2. Players gauge what an upgrade does by buying it and watching what happens — discovery, not explanation.

Concrete changes:

- `src/phase1/index.js`: remove the `upgradeDescriptions` object and the code path that renders descriptions in tooltips. Tooltip becomes: icon (already there) + cost in Roman (already there). Nothing else.
- `src/phase2/index.js`: remove the `Unlock <i data-lucide=...>...</i>` prefix in upgrade tooltips. Tooltip when locked: just the requirement icons. Tooltip when unlocked: just the cost.
- `src/phase2/index.js`: drop the `(industry: X.X)` parenthetical entirely from `starsPerPerson` rendering. The allocation slider already communicates the split visually.

The Bank tooltip in Phase 1 is already empty (v1.8.0 change). Stays empty.

### Rule for future

Add to CLAUDE.md design principles: **"Tooltips render icon and Roman cost only. No descriptive text. If a feature requires explanation, the feature is wrong, not the tooltip."**

## Part 4 — Phase 2 animation rework

The user's note: "Vi måste göra de här animationerna bättre ändå." Not just performance fixes — quality.

### Audit pass

Walk through every animated element in `style-stage2.css` and the inline Phase 2 animations. For each, ask:

1. Does the timing match the vision (calm, contained, one-shot for state changes, no UI loops except those that represent ongoing process like the factory)?
2. Does it survive idle states (does it pause when there's nothing to animate)?
3. Does it match the directional-clarity principle?
4. Is it readable on mobile?

### Known fixes (committed direction)

- **Factory conveyor** — currently loops `animation: p2-convey 3.2s linear infinite` even when the building has no production. Add `animation-play-state: paused` when `producesPerSecond === 0`. Resume on production.
- **Competitor island pulse** — the `enemy-pulse` keyframe loops forever once the island is visible. Pulse only during the fade-in (first 1.5 s after spawn at 40k pop), then settle to static. The chapter card at 50k provides the next dramatic beat.
- **Pop progress ring** (used on housing) — currently a `stroke-width: 2` ring (per the uncommitted local change). Verify it reads correctly across all building types and mobile sizes.
- **Mobile factory icon** — uncommitted local fix narrows the icon on <640 px. Commit this.

### Audit-driven fixes (will be discovered)

A first pass through Phase 2 in a browser, both desktop and mobile, with debug menu fast-forward. Document every animation that:

- Janks (frame drops, jumpiness)
- Restarts unexpectedly
- Plays before its parent is visible
- Doesn't survive a re-render

Each finding becomes a one-line task. Bundle the fixes into Part 4. Expected count: 3–8 small fixes. List goes into the implementation plan, not the design.

### Animation principle to add to CLAUDE.md

**"Animations that represent ongoing process (factory production, conveyor) loop only while the process is live. Animations for state changes (unlock, purchase, level-up) are one-shot. Decorative pulse loops on UI elements are forbidden — see vision §Tone."**

## Part 5 — Save/load hardening

### Current state

Both phases use `JSON.parse(localStorage.getItem(KEY))` directly. A corrupted value (manually edited, partially written, schema drift) crashes `init()` silently or shows blank UI.

### Target

- **Wrap every read in try/catch.** On parse failure: log to console, fall back to fresh state for that key, leave the key alone in localStorage so it can be inspected.
- **Schema version field.** Every save object grows a `schemaVersion: 1` field. Reads check it. If missing → treat as v1 (legacy). If higher than known → fall back to fresh and log. This sets up future migrations without breaking saves now.
- **Quota errors.** `localStorage.setItem` throws `QuotaExceededError` on full storage. Catch it. Per vision: silent fallback. The game continues running in memory; on next reload, the player's progress is gone but no error is shown.
- **Corruption is silent.** Per the user's explicit instruction. No toast, no badge, no "save was reset" message. The player simply finds themselves in a fresh state and moves on.

### Files

- `src/phase1/persistence.js` — already a module. Wrap `loadGameState` and `saveGameState`. Add schema version.
- `src/phase2/index.js` — save/load is inline. Move it into `src/phase2/persistence.js` (mirrors Phase 1). Apply same hardening.

### Tests

`__tests__/persistence.test.js` already exists for Phase 1. Extend with:

- Corrupted JSON → fresh state, original key preserved.
- Missing `schemaVersion` → treated as v1.
- Higher `schemaVersion` than known → fresh state.
- `QuotaExceededError` → save call returns false, doesn't throw.

Add equivalent tests for Phase 2 once it has its own persistence module.

## Part 6 — Playtest validation

A live walk-through to catch what static reading missed.

### Method

Use Playwright (already available in the dev environment via `.playwright-mcp/`). Three scenarios:

1. **Fresh Phase 1.** Clear localStorage. Open the game. Note: does the opening chapter card play? Is the first interaction obvious enough that *discovery* (not confusion) takes hold? Play through to Bank.
2. **Fresh Phase 2.** Continue from scenario 1 — chapter card I→II should fire on Bank click. Play Phase 2 from scratch up to ~1k pop. Note any unlock that arrives without a tease, any animation glitch, any tooltip wart.
3. **Endgame fast-forward.** Use debug menu to jump to 40k → see competitor spawn. Continue to 50k → see III·WAR card → "to come" wall. Verify the wall is final (no further input registers).

### Mobile pass

Same three scenarios at viewport widths **320 px, 375 px, 1024 px**. Capture screenshots, file mobile-specific bugs.

### Output

Each issue: one-line description, file:line if known, severity. Bundle into the implementation plan, not the design. Expected count: 5–15.

## Architecture summary

The release introduces one new module (`chapterCard.js`) and one refactor (`phase2/persistence.js` extracted from inline). It modifies four existing files significantly (`phase1/index.js`, `phase2/index.js`, `style.css`, `style-stage2.css`) and touches several others lightly (`main.js`, `index.html`, `CLAUDE.md`, `README.md`, `PROJECTPLAN.md`, `CHANGELOG.md`, `LESSONS.md`, `INDEX.md`).

There is no framework introduced, no build step added, no dependency changes beyond a Google Fonts `@import` for Bebas Neue. The architecture pattern (modular phases, AbortController-based teardown, ES modules served directly) is preserved.

## Risk and rollback

- **Chapter card on game start.** Risk: blocks all input for ~2.2 s before the game is interactive. Players who already played might be confused if they see it again. Mitigation: skip it for any save that already exists (`localStorage` check).
- **State transfer during card hold.** Risk: if `onMidpoint` throws, the card still finishes playing but the game is in an inconsistent state. Mitigation: try/catch in playChapterCard wraps the callback; failure logs but doesn't break the animation.
- **40k pop competitor spawn.** Risk: existing saves at >40k pop won't fire the spawn animation (already past threshold). Mitigation: spawn check uses `>= 40000`, so it fires immediately on load and the island fades in from 0 over 1.5 s. Same player experience.
- **GitHub Pages deployment.** Risk: CDN-loaded resources might be blocked on Pages domain. Mitigation: verification step in Part 8.
